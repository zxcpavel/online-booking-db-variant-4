## 🏗 Архитектура уровня доступа к данным

### Выбранный паттерн: Repository
Паттерн **Repository** был выбран для реализации DAL, так как он:
- Инкапсулирует логику доступа к данным в отдельных классах
- Предоставляет коллекционный интерфейс для работы с сущностями
- Отделяет бизнес-логику от логики работы с БД
- Упрощает тестирование и замену источника данных

## 🔧 Реализованные методы репозиториев

### AbstractRepository (базовый класс)

| Метод | Назначение | Параметры |
|-------|------------|-----------|
| `findAll()` | Выборка всех записей с фильтрацией | `$where` (массив условий), `$orderBy` (сортировка), `$limit` (лимит) |
| `findById($id)` | Поиск по первичному ключу | `$id` (идентификатор) |
| `insert($data)` | Вставка новой записи | `$data` (ассоциативный массив) |
| `update($id, $data)` | Обновление записи | `$id`, `$data` (поля для обновления) |
| `delete($id)` | Удаление записи | `$id` (идентификатор) |

### LessonRepository (специфичные методы)

| Метод | Назначение | Пример использования |
|-------|------------|---------------------|
| `getLessonsByDate($date)` | Получить все уроки на дату | Расписание на день |
| `getTeacherLoad($teacherId, $start, $end)` | Нагрузка преподавателя за период | Аналитика загруженности |
| `createLessonWithValidation(...)` | Создание урока с проверкой бизнес-правил | Запись ученика с транзакцией |
